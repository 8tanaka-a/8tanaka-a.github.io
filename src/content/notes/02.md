バックエンドの技術選定で、NestJS と Drizzle ORM の組み合わせに落ち着きました。
アプリの要件を、構造的に満たせるかどうかが判断基準でした。

NestJS は、DI のリクエストスコープでコンテキストを透過的に扱える。
Guard と Repository の基底クラスでデータの分離を強制できるので、実装漏れが構造上起きにくい。
この「忘れても壊れない」設計ができる点が、一番の決め手でした。

Drizzle は、SQL に近い記述で発行クエリが読める。
RLS の SET LOCAL 運用やデバッグがしやすく、Prisma のような Query Engine 起因の接続プーリング問題もない。
要件との相性が良いと感じています。

TypeScript で FE/BE を統一できる恩恵はすでに実感済み。
今回はその前提の上で、構造の強さで選びました。
